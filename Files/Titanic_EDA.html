Titanic Kaggle challenge (EDA)¶

The Titanic challenge is a classic introductory machine learning challenge in the Data Science universe.

The competition is simple: use machine learning to create a model that predicts which passengers survived the Titanic shipwreck. The sinking of the Titanic is one of the most infamous shipwrecks in history.

On April 15, 1912, during her maiden voyage, the widely considered unsinkable RMS Titanic sank after colliding with an iceberg. Unfortunately, there werent enough lifeboats for everyone on board, resulting in the death of 1502 out of 2224 passengers and crew. While there was some element of luck involved in surviving, it seems some groups of people were more likely to survive than others.

I have divided the problem into exploratory data analysis (EDA), and machine learning model development. This notebook addresses the EDA.

Kaggle provides a brief data description that is here reproduce:

Variable Definition Key
survival Survival 0 = No, 1 = Yes
pclass Ticket class 1 = 1st, 2 = 2nd, 3 = 3rd
sex Sex
Age Age in years
sibsp # of siblings / spouses aboard the Titanic
parch # of parents / children aboard the Titanic
ticket Ticket number
fare Passenger fare
cabin Cabin number
embarked Port of Embarkation C = Cherbourg, Q = Queenstown, S = Southampton

Variable Notes

pclass: A proxy for socio-economic status (SES) 1st = Upper 2nd = Middle 3rd = Lower

age: Age is fractional if less than 1. If the age is estimated, is it in the form of xx.5

sibsp: The dataset defines family relations in this way... Sibling = brother, sister, stepbrother, stepsister Spouse = husband, wife (mistresses and fiancs were ignored)

parch: The dataset defines family relations in this way... Parent = mother, father Child = daughter, son, stepdaughter, stepson Some children travelled only with a nanny, therefore parch=0 for them.

As seen most of the fields are numeric except Name, Sex, Ticket, Cabin and Embarked. From the numeric, only Age and Cabin are floats, presumably because they have missing values.

The number of unique values per column can through some light on how informative for a predictive model none numeric fields will be. Generally, the fact that the number of unique values is much lower than the total number of records is a good sign, because it allows for instances to be grouped or clustered by these shared identifiers. Hopefully, most of these characteristics will be correlated with the variable we are trying to predict.

def Age_ranges(x):
    if x < 18:
        return 'Child'
    elif x < 65:
        return 'Adult'
    elif x >= 65:
        return 'Elderly'

It would be useful to have plots that showed the survived/deceased split for the values of the different variables. A bar chart is one way to do this.

From the plots created by the cell below we can see that ages around 20 and 40 are more like to have deceased during the tragedy.

Passenger class 3 was by the most numerous class and the class that saw a higher number of deceased. It's somewhat interesting that the number of people that survived of each class is relatively similar.

The relative number of deceased for Mr. was much higher as it is an indicator of a male passenger, while Miss and Mrs had higher numbers or survived compared to deceased.

As mentioned previously, even though the number of males on board was almost more than double the number of females, the number of males survivers was less than have that of females.

Curiously people with one sibling on board were substantially more likely to survive.

People travelling with no family members were sadly more likely to decease.

We must be cautious to draw any conclusions from the cabin data because the sample is very low. But it appears that B cabin passengers were much more likely than C or G passengers to survive.

Passengers embarked at Cherbourg were more likely to survive. Upon further inspection, it was observed that the relative percentage survival of males embarked at Cherbourg (France) was higher than that of their counterparts embarked at Queenstown or Southampton.
